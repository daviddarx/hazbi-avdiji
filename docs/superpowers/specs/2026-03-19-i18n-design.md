# Internationalization (i18n) Design Spec

## Overview

Add multilingual support to the Hazbi Avdiji website. French remains the primary language, English is added now, German may follow later. Content is managed per-locale in TinaCMS using folder-based localization. Users switch language via a UI component matching the existing dark mode switcher pattern.

## Locales

- `fr` — French (default)
- `en` — English
- Future: `de` — German

## URL Structure

Sub-path routing with **all locales prefixed** (including the default):

- `/fr/idees-recues` — French page
- `/en/misconceptions` — English page
- `/` redirects to `/fr/`

Each locale can have its own slugs (translated). The mapping between locales is explicit via a `translationOf` reference field in TinaCMS.

## Next.js Configuration

### `next.config.js`

```js
module.exports = {
  i18n: {
    locales: ['fr', 'en'],
    defaultLocale: 'fr',
    localeDetection: false,
  },
  // ... existing config
};
```

**Note:** Next.js Pages Router i18n with `defaultLocale: 'fr'` means `/page` serves French without prefix. Since we want `/fr/page` always, we add a middleware or redirect from `/page` to `/fr/page` for all non-`/en` paths. Alternatively, we handle this with a root redirect from `/` to `/fr/` and accept that both `/page` and `/fr/page` serve French (with canonical URL set to `/fr/page` in meta tags).

**Recommended approach:** Use Next.js rewrites/redirects to ensure `/fr/` prefix is always present in the URL, and set `<link rel="canonical">` to the `/fr/` version. This avoids fighting the framework's default behavior.

## TinaCMS Schema Changes

### New `translationOf` Reference Field

Added to `page`, `post`, and `category` collections. This field links a document to its counterpart in another locale. It is a reference to the same collection.

```ts
{
  name: 'translationOf',
  label: 'Translation of',
  type: 'reference',
  collections: ['page'], // or 'post', 'category' respectively
  description: 'Link to the corresponding page in the other language',
}
```

### Collection Path Updates

Each collection's `path` is updated to include the locale subfolder. Since TinaCMS doesn't natively support locale-aware paths, we use **one collection per locale per content type**, or we keep a single collection with the broader path and rely on the folder structure.

**Recommended approach:** Keep single collections with the existing paths. The locale is part of the `relativePath` when querying (e.g., `fr/home.mdx`). The folder structure within `content/pages/` naturally separates locales. This minimizes schema changes and keeps TinaCMS admin simple — the editor sees all locales' files in one list, organized by folder.

### Updated `getStaticProps`

```ts
export const getStaticProps = async ({ params, locale }) => {
  const pageMdPath = params.slug ? params.slug[0] : 'home';

  const pageResult = await client.queries.page({
    relativePath: `${locale}/${pageMdPath}.mdx`,
  });

  const navigationResult = await client.queries.navigation({
    relativePath: `${locale}/navigation.md`,
  });

  // ... same pattern for footer, posts, categories
};
```

### Updated `getStaticPaths`

```ts
export const getStaticPaths = async ({ locales }) => {
  const pageConnectionResult = await client.queries.pageConnection();
  const paths = [];

  pageConnectionResult.data.pageConnection.edges.map((edge) => {
    const fileName = edge.node._sys.filename;
    const filePath = edge.node._sys.path; // e.g., "content/pages/fr/home.mdx"
    const locale = filePath.split('/')[2]; // extract locale from path

    paths.push({
      params: { slug: [fileName] },
      locale,
    });
  });

  // Add home paths for each locale
  locales.map((locale) => {
    paths.push({ params: { slug: [''] }, locale });
  });

  return { paths, fallback: 'blocking' };
};
```

Same pattern applies to `src/pages/idee-recue/[slug].tsx` for posts.

## Language Switcher Component

### `LanguageSwitcher.tsx`

Follows the same pattern as `DarkModeSwitcher.tsx`:

- Uses HeadlessUI `Menu` / `MenuButton` / `MenuItems`
- Framer Motion `AnimatePresence` for open/close animation
- `getMenuMotionVariants()` for responsive animation
- Displays a globe/language icon in the button
- Dropdown shows "Français" and "English" options
- Active locale is highlighted (same `bg-themed-next` pattern)
- Stores locale preference in `localStorage` for future visits

### Resolving the Translation Link

When the user clicks to switch language, the component:

1. Reads the current page's `translationOf` field from the page props
2. Extracts the target locale's slug from the referenced document
3. Navigates to `/${targetLocale}/${targetSlug}`

If no `translationOf` is set, falls back to the homepage of the target locale.

### Placement in Layout

In `Layout.tsx`, placed next to `DarkModeSwitcher`:

```tsx
<div className="flex justify-end gap-12 ...">
  {navigationProps && <Navigation {...navigationProps} />}
  <LanguageSwitcher translationOf={translationOfProps} />
  <DarkModeSwitcher />
  {navigationProps && <MobileNavigation {...navigationProps} />}
</div>
```

The existing `content/translations.ts` becomes locale-keyed:

```ts
const t = {

    logoTitle: 'Dr. Hazbi Avdiji',
    logoSubline: 'Votre généraliste en diversité et inclusion',
    allPosts: 'Tout',
    post: 'Idée reçue',
    postResults: (amount: number) => (amount === 1 ? 'Idée reçue' : 'Idées reçues'),
    // ... all existing strings
  },
  en: {
    logoTitle: 'Dr. Hazbi Avdiji',
    logoSubline: 'Your generalist in diversity and inclusion',
    allPosts: 'All',
    post: 'Misconception',
    postResults: (amount: number) => (amount === 1 ? 'Misconception' : 'Misconceptions'),
    // ... all translated strings
  },
};
```

### `useTranslations()` Hook

A custom hook that reads the current locale from `useRouter().locale` and returns the corresponding string set:

```ts
import t from '@/content/translations';
import { useRouter } from 'next/router';

export function useTranslations() {
  const { locale } = useRouter();
  return t[locale as keyof typeof t] || t.fr;
}
```

All components currently importing `t` from `@/content/translations` are updated to use `useTranslations()` instead.

## SEO Considerations

- `<html lang>` attribute set to current locale
- `<link rel="alternate" hrefLang="fr" href="/fr/...">` and `<link rel="alternate" hrefLang="en" href="/en/...">` on all pages
- `<link rel="canonical">` always points to the `/fr/` or `/en/` prefixed URL
- Meta descriptions translated per locale

## Post Route

The `postRoute` constant in `src/utils/tina.ts` becomes locale-aware:

```ts
export const postRoutes = {
  fr: '/idee-recue',
  en: '/misconception',
};
```

This means the English post detail route is `/en/misconception/[slug]` and the French route remains `/fr/idee-recue/[slug]`. The `src/pages/idee-recue/[slug].tsx` file needs to handle both, or we create a second route file at `src/pages/misconception/[slug].tsx`. Since Next.js i18n routing prefixes with locale but keeps the same page file, we may need to use a catch-all or a rewrite to handle translated route segments.

**Recommended approach:** Keep a single `src/pages/idee-recue/[slug].tsx` and use Next.js rewrites to map `/en/misconception/:slug` to `/en/idee-recue/:slug` internally. This avoids duplicating page files while keeping clean English URLs.

```js
// next.config.js rewrites
async rewrites() {
  return [
    { source: '/admin', destination: '/admin/index.html' },
    { source: '/en/misconception/:slug', destination: '/en/idee-recue/:slug' },
  ];
}
```
